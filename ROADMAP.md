# Roadmap

> Generated by /roadmap on 2026-06-21 (refresh, autopilot batch item 5). Source of truth is
> docs/backlog/ - edit items there, then re-run /roadmap. Manual edits may be overwritten.

Relay is a distributed task/render-farm orchestrator: three Go binaries (relay-server =
HTTP + gRPC + scheduler with an embedded React/Vite SPA; relay-agent worker that holds one
gRPC stream and runs each task as a subprocess; the relay CLI), plus an MCP server and a
Perforce source-provider. The current phase is hardening. A 2026-06-10 internal codebase
review surfaced a class of concurrency/correctness bugs that sidestep the cross-cutting
invariants now documented in CLAUDE.md - epoch fencing on every task-status write, one
bounded sender per gRPC stream, identity-checked teardown, the single job-spec pipeline,
no interior pointers across locks, and the single JSON entry point. The team is closing
those findings one at a time; recent sessions shipped the requeue/retry epoch bump, the
stale-stream-teardown fix, the nil-provider source guard, cron source-spec persistence, the
Perforce registry race fix, the runner pipe-drain hang fix, the job status recompute race
fix, the forced-cancel send-backpressure fix, the archived-user offboarding fix, the
background-sweeper Prepare TOCTOU fix, the MCP structured-error delivery fix, and the CLI
admin/profile stdout fix, with priority on correctness of the thing being operated - task
dispatch, scheduling, and source workspaces - before broadening features. A secondary
workstream builds out the embedded web SPA (auth, Workers, Jobs, and Schedules lists shipped;
detail pages, list filters, and Admin/Profile still pending). MCP, CLI, and store/schema work
fill in behind those two.

- [useJobs and useJobStats share query-key prefix](docs/backlog/closed/bug-2026-06-05-usejobs-usejobstats-query-key-prefix.md) - closed 2026-06-21; `useJobStats`'s react-query key changed from `['jobs', 'stats']` to a distinct top-level `['job-stats']`, so a broad `invalidateQueries({ queryKey: ['jobs'] })` no longer clobbers the stats query. A grep confirmed the definition was the only reference site. New `queryKeyDecoupling.test.tsx` proven RED (stats refetched 2x vs expected 1x); full web suite (149) + `tsc` green.
- [formatRelativeTime duplicated across workers and schedules modules](docs/backlog/closed/bug-2026-06-05-formatrelativetime-duplicated.md) - closed 2026-06-21; the verbatim-identical `formatRelativeTime` was extracted into a new shared `web/src/lib/time.ts`, and `web/src/workers/liveness.ts` + `web/src/schedules/format.ts` now re-export it so all seven importers keep their import paths. Behavior unchanged (the two bodies were character-for-character identical, confirmed before merging). New `web/src/lib/time.test.ts`; full web suite (148 tests) + `tsc --noEmit` green.
- [MCP create_schedule overlap_policy description says "queue" but value is rejected](docs/backlog/closed/bug-2026-06-20-mcp-overlap-policy-description-says-queue.md) - closed 2026-06-21; the `relay_create_schedule` `overlap_policy` jsonschema description was corrected from "skip or queue" to "skip or allow", matching the only two accepted values (the API handler's `skip`/`allow` validation and migration 000019's `scheduled_jobs_overlap_policy_check`). Pure doc-string fix; the `relay_update_schedule` arg named no specific value and was untouched. Build + vet clean.
- [relay admin users list/get and profile update print data output to stderr](docs/backlog/closed/bug-2026-06-10-admin-output-stderr.md) - closed 2026-06-21; `AdminCommand`/`ProfileCommand` wired `stderrWriter()` as the single output writer for all subcommands, so `relay admin users list/get` tables and `relay profile update` results went to stderr (`> users.txt` captured nothing, `| grep` failed). Routed data output to `os.Stdout` (mirroring the jobs/workers pattern) while keeping the password prompts in `doAdminPasswd`/`doAdminUsersCreate` explicitly on `stderrWriter()`. `printUserDetail` (shared by create/update/archive) also correctly moves to stdout. Three tests drive the real `AdminCommand()`/`ProfileCommand()` constructors with `os.Stdout`/`os.Stderr` redirected to pipes - the inner-writer-injection tests could not have caught the constructor wiring bug - proven RED against the old wiring. Combined code review returned no findings.
- [MCP structured errors (code/hint) never reach clients](docs/backlog/closed/bug-2026-06-10-mcp-structured-errors-lost.md) - closed 2026-06-21; a generic `addTool[A, R]` registration wrapper now returns an `IsError` `CallToolResult` carrying the marshaled `ToolError` (code/message/hint) instead of returning the error to the go-sdk, which flattened it to `code: message` text and dropped the hint. All 18 tool registrations across 12 files were migrated to the wrapper. Two delivery tests over the in-memory MCP transport (401 + validation), the 401 case RED-proven.
- [Web SPA models job statuses the backend never emits](docs/backlog/closed/bug-2026-06-20-web-job-status-vocabulary-drift.md) - closed 2026-06-21; the SPA `JobStatus` union was narrowed to the real `jobs.status` vocabulary, dead `statusColor`/`JobsTable` cases removed, and the Jobs "Queued" filter chip now sends `status=pending` so it matches jobs instead of the never-matching `status=queued`.
- [relay logs/submit hang forever if the job goes terminal before the SSE subscribe](docs/backlog/closed/bug-2026-06-10-cli-logs-terminal-race-hang.md) - closed 2026-06-21; `watchJobLogs` was restructured from GET-first-then-subscribe to subscribe-first-then-snapshot via a new `onSubscribed` hook, so a job that went terminal before the subscribe is caught instead of hanging on an event-less stream.
- [SPA pagination corrupts the cursor stack when next is clicked during a fetch](docs/backlog/closed/bug-2026-06-10-spa-pagination-cursor-corruption.md) - closed 2026-06-21; JobsPage and SchedulesPage now disable both pagination buttons while `isPlaceholderData` is true, closing the double-next-during-fetch window that stranded a later prev.
- [Brief inconsistent-state window on worker re-enroll](docs/backlog/closed/bug-2026-06-05-inconsistent-state-window-reenroll.md) - closed 2026-06-21; `SetWorkerAgentToken` now clears the revoked status atomically with `revoked_at`, so a re-enrolling worker never sits in the `status='revoked'` + `revoked_at=NULL` window.
