Relay is a distributed task/render-farm orchestrator: three Go binaries (relay-server =
HTTP + gRPC + scheduler with an embedded React/Vite SPA; relay-agent worker that holds one
gRPC stream and runs each task as a subprocess; the relay CLI), plus an MCP server and a
Perforce source-provider. The current phase is hardening. A 2026-06-10 internal codebase
review surfaced a class of concurrency/correctness bugs that sidestep the cross-cutting
invariants now documented in CLAUDE.md - epoch fencing on every task-status write, one
bounded sender per gRPC stream, identity-checked teardown, the single job-spec pipeline,
no interior pointers across locks, and the single JSON entry point. The team has now closed
that review's agent/runner concurrency findings one at a time; recent sessions shipped the
requeue/retry epoch bump, the stale-stream-teardown fix, the nil-provider source guard, cron
source-spec persistence, the Perforce registry race fix, the runner pipe-drain hang fix, the
job status recompute race fix, the forced-cancel send-backpressure fix, the archived-user
offboarding fix, the background-sweeper Prepare TOCTOU fix, the MCP structured-error delivery
fix, the CLI admin/profile stdout fix, the cancel/disable concurrent-fan-out fix, the
default-cancel/Abandon unbounded-hang fix, the default-cancel sendInventory wedge-escape, the MCP
discovery-time role filtering for admin-only tools, the MCP token mid-session reload, the
per-eviction `p4 client -d` timeout (`RELAY_EVICTION_TIMEOUT`), the `relay_wait_for_job`
adaptive poll, and the MCP resource templates (`relay://jobs/{id}`, `relay://tasks/{id}`) plus the
`relay://recent-jobs` TTL cache (`RELAY_MCP_RESOURCE_CACHE_TTL`), with priority on correctness of
the thing being operated - task dispatch, scheduling, and source workspaces - before broadening
features. With the agent/runner concurrency-bug stream drained, the MCP bug queue fully closed, and
the source-workspace eviction hardened, the hardening phase is largely complete; attention now
shifts to the embedded web SPA build-out (auth, Workers, Jobs, and Schedules lists shipped; detail
pages, list filters, and Admin/Profile still pending), with remaining MCP/agent ideas, CLI, and
store/schema work filling in behind it.

## Suggested backlog actions

Cross-referencing `design_handoff_relay_holo/` against the code surfaced five front-end surfaces the
Holo handoff specifies that have no backlog item yet. They are the bulk of the remaining front end;
file each with `/backlog`:

- **Admin console pages (feature).** The whole `/admin` route is a `JobsPlaceholder` stub (`web/src/app/router.tsx:24`) with no UI. Holo specifies five tabs - Users, Invites, Agent enrollments, Reservations, Server. Most backend exists and is admin-gated (users CRUD, `POST /v1/invites`, agent-enrollments, reservations), so Users / Enrollments / Reservations tabs are frontend-only; the Invites tab also needs a new `GET /v1/invites` list endpoint. Largest single remaining surface - consider slicing per tab.
- **Profile pages (feature).** `/profile/*` is also a stub (`web/src/app/router.tsx:25`) yet UserMenu already links to /profile, /profile/password, /profile/sessions. Holo specifies Identity / Password / Sessions tabs. Identity (`PATCH /v1/users/me`) and Password (`PUT /v1/users/me/password`) are frontend-only; the Sessions list needs a new `GET /v1/auth/tokens` endpoint ("sign out everywhere" via `DELETE /v1/auth/tokens` already exists).
- **Task log view + live SSE tailing (feature; needs a decision).** Holo's full-screen `HoloTaskLog` (`/jobs/:id/tasks/:n`) streams a task log live, and the job-detail page needs the same. There is no log-streaming UI and no `?follow=1` on the task-logs endpoint; decide between consuming the existing `GET /v1/events?job_id=` SSE (present) and adding `?follow=1`. This is the SPA's first EventSource client and a prerequisite for the job-detail live-log tab.
- **Job actions: submit / cancel / retry (feature).** The SPA has no job write-actions. Holo shows "+ New job" (a job-spec submit form) and graceful + force cancel on job detail. Submit (`POST /v1/jobs`) and cancel (`DELETE /v1/jobs/{id}?force=`) are frontend-only; retry needs a new `POST /v1/jobs/{id}/retry` route, which would also reactivate [jobs-stats-24h-updated-at-proxy](docs/backlog/bug-2026-06-05-jobs-stats-24h-updated-at-proxy.md).
- **Extract shared Holo primitives (idea).** The theme tokens already match the handoff, but glass / eyebrow / chip styling is re-inlined per page. Extract a small shared set (GlassPanel, Eyebrow, Chip, KPI stat, lift StatusDot to shared) before the Admin / Profile / job-detail / schedule-detail surge to avoid divergence. Pairs with the accessible-table primitive item. Frontend-only, small.
- Consider filing a feature item for the real prerequisite behind
  [remove-synthetic-step-marker](docs/backlog/idea-2026-04-26-remove-synthetic-step-marker.md): persist
  `step_index`/`step_total` onto `task_logs` and expose them on the logs API, so step boundaries have a
  structured form and the text marker can later be retired. Until then the step-marker deletion stays
  blocked.
