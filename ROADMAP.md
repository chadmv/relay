Relay is a distributed task/render-farm orchestrator: three Go binaries (relay-server =
HTTP + gRPC + scheduler with an embedded React/Vite SPA; relay-agent worker that holds one
gRPC stream and runs each task as a subprocess; the relay CLI), plus an MCP server and a
Perforce source-provider. The current phase is hardening. A 2026-06-10 internal codebase
review surfaced a class of concurrency/correctness bugs that sidestep the cross-cutting
invariants now documented in CLAUDE.md - epoch fencing on every task-status write, one
bounded sender per gRPC stream, identity-checked teardown, the single job-spec pipeline,
no interior pointers across locks, and the single JSON entry point. The team has now closed
that review's agent/runner concurrency findings one at a time; recent sessions shipped the
requeue/retry epoch bump, the stale-stream-teardown fix, the nil-provider source guard, cron
source-spec persistence, the Perforce registry race fix, the runner pipe-drain hang fix, the
job status recompute race fix, the forced-cancel send-backpressure fix, the archived-user
offboarding fix, the background-sweeper Prepare TOCTOU fix, the MCP structured-error delivery
fix, the CLI admin/profile stdout fix, the cancel/disable concurrent-fan-out fix, the
default-cancel/Abandon unbounded-hang fix, the default-cancel sendInventory wedge-escape, the MCP
discovery-time role filtering for admin-only tools, and the MCP token mid-session reload, with
priority on correctness of the thing being operated - task dispatch, scheduling, and source
workspaces - before broadening features. With the agent/runner concurrency-bug stream now drained
to ideas and the MCP auth bugs closed, attention is on the remaining MCP fidelity items, the
source-workspace robustness ideas, and the embedded web SPA build-out (auth, Workers, Jobs, and
Schedules lists shipped; detail pages, list filters, and Admin/Profile still pending). CLI and
store/schema work fill in behind those.

## What moved

- **Shipped + closed: [mcp-token-mid-session-expiry](docs/backlog/closed/bug-2026-05-09-mcp-token-mid-session-expiry.md).** The prior Now #1 / mcp #1 bug landed through the full lifecycle: `relayclient.Client`'s token is now `RWMutex`-guarded with `SetToken`/`Token`, and a single `Server.do` chokepoint (routing all ~21 MCP tool calls) reloads the token from config on a 401 and retries exactly once when the reloaded token is non-empty and differs from the one the call used, so `relay login` mid-session takes effect without a client restart. Steady-state calls do zero extra I/O. `-race` clean in Docker; integration green; clean review.
- **Reordered: eviction-timeout and wait-for-job promoted.** With the MCP auth bug stream (`role-filtering` + `token-reload`) closed, `eviction-timeout` (source-workspace robustness) and `relay_wait_for_job-poll-interval` move up into Now, joined by `mcp-resources-caching-templates`; `p4d-binary-checksum`, `job-detail-page`, and `jobs-stats-24h-proxy` fill Next. `worker-detail-activity-panel` surfaces into Later as the next substantive web-frontend slice now that the bug queue is thinning.

## Recently shipped

- [MCP token read once at startup - mid-session expiry requires client restart](docs/backlog/closed/bug-2026-05-09-mcp-token-mid-session-expiry.md) - closed 2026-06-25; `relayclient.Client`'s token is now `sync.RWMutex`-guarded with `SetToken`/`Token` accessors; a single `Server.do` chokepoint (all ~21 MCP tool calls route through it) reloads the token from config on a 401 and retries exactly once when the reloaded token is non-empty and differs from the token the call used, so `relay login` mid-session takes effect without a client restart. A second 401 still surfaces `auth_expired`; steady-state calls do zero extra I/O. A concurrency fix (compare against the token the call used, not the live one) was found under `-race`. Unit + on-disk reload + negative-case tests, `-race` clean in Docker, integration green; review found no issues.
- [MCP admin-only tools have no discovery-time role filtering](docs/backlog/closed/bug-2026-05-09-mcp-admin-tools-role-filtering.md) - closed 2026-06-25; `NewServer` resolves caller identity once at startup via `callWhoami` and registers admin-only `relay_list_reservations` only when `is_admin == true` (fail-closed comma-ok assertion); whoami failure at startup returns the error from `NewServer` (clean exit, `cli/mcp.go` untouched). Discovery filtering is cosmetic - the server-side `auth(admin(...))` gate plus `forbidden` ToolError fallback stay authoritative and unchanged. New unit tests over the real SDK `ListTools` surface + integration assertions; ~16 MCP test files repaired via a shared `whoamiHandler`/`newWhoamiBackend` helper. Unit + Docker + integration green; review found no issues.
- [sendInventory in the default-cancel defer can park unbounded under a wedged sendCh](docs/backlog/closed/bug-2026-06-21-sendinventory-blocking-send-under-wedge.md) - closed 2026-06-25; `sendInventory` now uses a room-first bounded best-effort try-send (`select { case r.sendCh <- msg: default: }`, mirroring `sendFinalStatus`) when `r.cancelled.Load() || r.abandoned.Load()`, so the deferred cleanup goroutine no longer parks until agent shutdown under a wedged `sendCh`; normal completion stays blocking. Three `//go:build !windows` tests RED-vs-GREEN on Linux/Docker; `internal/agent` suite + `go vet` clean; review found no high/medium/low issues. Closes residual #3 of the 2026-06-21 default-cancel/Abandon hang design.
- [Default cancel / Abandon() hang unbounded under a wedged sendCh](docs/backlog/closed/bug-2026-06-20-default-cancel-abandon-backpressure-bound-test.md) - closed 2026-06-21 (PR #83); per-runner `cancelledCh` gives the default `Cancel(false)` / `Abandon()` log-write and terminal-status paths a per-task escape so a wedged `sendCh` no longer hangs `cmd.Wait()` unbounded.
- [Through-the-stack re-enrollment integration test](docs/backlog/closed/idea-2026-06-05-reenrollment-integration-test.md) - closed 2026-06-21; `TestConnect_EnrollmentTokenRevivesRevokedWorker` drives a revoked worker back through the gRPC enrollment-token revive path.
