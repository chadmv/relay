# Roadmap

> Generated by /roadmap on 2026-06-20 (refresh). Source of truth is docs/backlog/ - edit
> items there, then re-run /roadmap. Manual edits to this file may be overwritten.

Relay is a distributed task/render-farm orchestrator: three Go binaries (relay-server =
HTTP + gRPC + scheduler with an embedded React/Vite SPA; relay-agent worker that holds one
gRPC stream and runs each task as a subprocess; the relay CLI), plus an MCP server and a
Perforce source-provider. The current phase is hardening. A 2026-06-10 internal codebase
review surfaced a class of concurrency/correctness bugs that sidestep the cross-cutting
invariants now documented in CLAUDE.md - epoch fencing on every task-status write, one
bounded sender per gRPC stream, identity-checked teardown, the single job-spec pipeline,
no interior pointers across locks, and the single JSON entry point. The team is closing
those findings one at a time; the last several sessions shipped the requeue/retry epoch
bump, the stale-stream-teardown fix, the nil-provider source guard, cron source-spec
persistence, the Perforce registry race fix, the runner pipe-drain hang fix, the job
status recompute race fix, the forced-cancel send-backpressure fix, and the archived-user
offboarding fix (token rejection + schedule disable), with priority on correctness of the
thing being operated - task dispatch, scheduling, and source workspaces - before
broadening features. A secondary workstream builds out the embedded web SPA (auth,
Workers, Jobs, and Schedules lists shipped; detail pages, list filters, and Admin/Profile
still pending). MCP, CLI, and store/schema work fill in behind those two.

## Dispatch, epoch & stream-teardown races (dispatch-races)

1. [Agent send goroutine not awaited across reconnect](docs/backlog/bug-2026-06-18-agent-reconnect-send-goroutine-not-awaited.md) (bug, medium) - `connect()` returns without joining its send goroutine, so a reconnect can leave two goroutines draining one `sendCh` and silently drop a queued message. Violates the one-bounded-sender invariant.
2. [Reconnect backoff never resets in agent and NotifyListener](docs/backlog/bug-2026-06-10-reconnect-backoff-never-resets.md) (bug, medium) - the backoff reset is unreachable on the drop path, so reconnect delay degrades monotonically toward the 60s cap and dispatch latency falls back to the 30s safety poll.
3. [Cancel/disable handlers send synchronously to workers](docs/backlog/bug-2026-06-10-cancel-disable-handlers-send-synchronously.md) (bug, low) - bounded after the wedged-agent fix, but a multi-task job on one wedged worker can still take up to N x ~5s in the request path.
4. [Brief inconsistent-state window on worker re-enroll](docs/backlog/bug-2026-06-05-inconsistent-state-window-reenroll.md) (bug, low) - between the enroll tx clearing `revoked_at` and the post-commit status flip, a worker momentarily appears revoked with a null timestamp.

## Recently shipped

- [Dispatch failure paths are inconsistent and silent](docs/backlog/closed/bug-2026-06-10-dispatch-failure-paths-silent.md) - closed 2026-06-20; a shared `failClaimedTask` helper terminally fails a claimed task for both bad-JSON cases - reusing the epoch-fenced `UpdateTaskStatus` at the claim's own non-zero epoch (not bumped, since `failed` is terminal), cascading `FailDependentTasks`, recomputing job status, and publishing `task` + `job` SSE events. This ends the dispatched-slot leak on bad `source` and the infinite claim/requeue churn (unbounded `assignment_epoch`) on bad `commands`. Every DB error path in `dispatch`/`handleTaskStatus` now logs via `log.Printf`; the benign `ClaimTaskForWorker` `pgx.ErrNoRows` claim race stays silent. New scheduler tests cover no-requeue, epoch stability, no-slot-leak, and the terminal job event.

One-line change in `web/src/auth/AuthProvider.tsx` plus an AuthProvider regression test that seeds the cache, logs out, and asserts it is empty.
